# Skill 文档编写范式

本文件是写 Shortcut Skill 文档的模板。新增任何带枚举参数的 Shortcut 时，照此范式写。

## 三层冗余原则

枚举参数（状态、分类、类型等）必须在三个位置各写一遍映射，确保 AI 无论从哪里进入都能命中：

### 第 1 层：CLI flag Desc 内联映射

Go 代码里 flag 的 `Desc` 字段，用 `值含义/值含义/值含义` 格式一行写完。

```
{Name: "work-order-status", Desc: "待办状态: 1待分派/2待接单/3处理中/4待确认/5待回访（空=全部）"},
{Name: "handle-status",     Desc: "经办分类: 0我创建的/1我分派的/2我接单的/3我处理的/4我确认的/5我回访的（空=全部）"},
```

原则：AI 只看 `--help` 也能猜对参数值，不用翻文档。

### 第 2 层：skill 速查表

`skills/references/products/<product>.md` 对应 Shortcut 章节里，用三列表格：

```markdown
| 数值 | 后端含义 | 用户可能这么说 |
|------|---------|----------------|
| （不传） | 全部 | "我的工单"、"我的待办"、"我有什么工单"、"待办列表" |
| `1` | 待分派 | "待我分派的"、"没派出去的"、"还没分配的"、"等我派单的" |
| `2` | 待接单 | "等我接的"、"派给我但还没接的"、"待我接单的"、"新派给我的" |
```

原则：AI 看到用户口语 → 在"用户可能这么说"列模糊匹配 → 拿到数值。
每行给 4-5 种口语变体，覆盖最常见的说法。

### 第 3 层：用户口语 → 命令句式

同章节里，用 `用户说"X" → 命令` 一行一条，做正向意图匹配：

```markdown
**用户口语 → 命令示例**：
- 用户说"看下我的工单 / 我的待办" → `workorder +list`
- 用户说"我正在处理的工单" → `workorder +list --work-order-status 3`
- 用户说"等我确认的单子" → `workorder +list --work-order-status 4`
- 用户说"新派给我还没接的" → `workorder +list --work-order-status 2`
```

原则：AI 从用户原话做正向匹配，直接出命令，不用反向查表。

## 两个 Shortcut 有同名字段时的处理

如果不同 Shortcut 的参数用同样的数字但含义不同（如 `+list` 的 `1`=待分派 vs `+handle-list` 的 `1`=我分派的），**必须**加警告块：

```markdown
> ⚠️ 同样的数字，`+list` 和 `+handle-list` 含义完全不同：
> - `+list --work-order-status 1` = 待分派的待办
> - `+handle-list --handle-status 1` = 我分派过的经办
> 不要搞混。
```

## 文档结构模板

每个 Shortcut 章节按以下顺序组织：

```markdown
### +<command> — <中文标题> ✅/🚧

一句话说明 + 与相关命令的区别（如有）。

**<枚举参数名>速查（`--<flag-name>`）**：

| 数值 | 后端含义 | 用户可能这么说 |
|------|---------|----------------|
| （不传） | 全部 | "..." |
| `0` | ... | "..."、..." |

**用户口语 → 命令示例**：
- 用户说"..." → `...`
- 用户说"..." → `...`

> ⚠️ 如果跟其他命令的参数值有歧义，加警告块。

**参数**：

| 参数 | 必填 | 说明 |
|------|------|------|
| `--<flag>` | 否 | 枚举描述: 值含义/值含义/值含义（空=全部） |
| `--page` | 否 | 页码（默认 1） |
| `--page-size` | 否 | 每页条数（默认 10） |
| `--dry-run` | 否 | 只预览请求，不实际执行 |

> 自动注入的字段写在这里（如 `assignees`、`headId`、`topLevelId`）。

返回结构说明 + 向用户汇报时的格式要求。
```

## 自动注入 vs 用户传参

- **自动注入**：从 Token 中获取的字段（`topLevelId`、`assignees`、`headId`），用户/AI 不需要传，在文档里用 `>` 引用说明
- **用户传参**：直接透传后端字段名，flag 名用 kebab-case（如 `work-order-status` → `--work-order-status`），不要造友好名映射层

## 意图判断写法

`skills/references/intent-guide.md` 里用同样的三层范式写跨 Shortcut 的口语映射：

```markdown
**`+list`（我的待办）→ `--work-order-status`**：
- 用户说"我的待办 / 我的工单" → 不传（全部）
- 用户说"处理中的 / 我正在做的" → `3`

**`+handle-list`（我经办的）→ `--handle-status`**：
- 用户说"我经办的 / 我处理过的" → 不传（全部）
- 用户说"我创建的 / 我报修的" → `0`
```

## 新增枚举参数的检查清单

- [ ] Go flag Desc 内联映射（第 1 层）
- [ ] skill 速查表：三列（数值 / 后端含义 / 用户可能这么说），每行 4-5 种口语（第 2 层）
- [ ] 用户口语 → 命令句式（第 3 层）
- [ ] 与其他 Shortcut 有同名字段 → 加 ⚠️ 警告块
- [ ] intent-guide.md 同步更新
- [ ] dry-run 验证请求体字段名和值正确
